### [NVIDIA/cosmos](https://github.com/NVIDIA/cosmos) — NEW ⭐ 9,037（+133）| Jupyter Notebook | 信源：中

**它是什么**: NVIDIA 的"世界模型"开放平台——一套能生成逼真物理场景视频/数据的 AI 模型 + 数据集 + 工具。用大白话说：你想训练一个机器人或自动驾驶系统，但在真实世界采集训练数据又慢又贵又危险，cosmos 让你"凭空生成"大量符合物理规律的场景视频当训练素材。随 GTC Taipei 刚发布 Cosmos 3。

**解决什么问题**: 过去训练 Physical AI（机器人、自动驾驶）必须靠真机跑、真路测，采一条边角案例（比如行人突然冲出）成本极高。cosmos 让团队"不部署真机就能生成训练数据"，把数据采集从物理世界搬到仿真。

**谁在用、怎么用**（从 Issues/HN 提取真实场景）:
- 机器人开发者落地：#184 询问 egocentric / robot embodiment 的 action channel 与 normalizer 配置；#180 在 DGX Spark 上跑遇 pynvml 报错——都是真实使用产生的技术问题。
- 第三方采用佐证：Toyota Research 用 cosmos 跑出 3 项 SOTA，FieldAI / Skild AI 用于通用机器人"大脑"的数据生成。

**社区怎么评价**:
- 👍 正面: NVIDIA 官方账号矩阵 + 第三方 KOL（Tibor Blaho、Andrew Curran）强力推广；有真实企业采用背书。
- 👎 负面: HN（Cosmos 3 发布帖 149 分 / 28 评）批评集中——(1) 算力门槛高，Nano 需 RTX PRO 6000 工作站 + $3–5 万机器人硬件；(2) demo 质量差（仓库场景人物反应不自然、自动驾驶 demo 闯红灯）质疑训练数据；(3) 援引"Bitter Lesson"批评 two-tower 架构把知识硬编进 agent；(4) 质疑只能生成"已大量存在的 in-domain 数据"，对需要 edge case 的前沿团队价值有限。#183"无法复现 paibench-G 结果"未获回应。
- 📊 健康度: 9,037 stars / 89 watchers ≈ 102:1（大厂明星仓常态，非刷星）；open issues 仅 5 无积压；contributors 仅 10 人，NVIDIA 内部团队主导、外部贡献极少——"开放权重但封闭开发"。

**风险与争议**:
- **compliance**: License 为 OpenMDW-1.1（NVIDIA 自定开放模型许可），非标准 OSI，商用条款需审查。
- **cost**: 算力成本高——Super 64B 需 Hopper/Blackwell，Nano 16B 需工作站级 GPU，edge 版"即将推出"未交付。
- **dependency**: backbone 用 Qwen3-VL 权重初始化，衍生/合规关系需注意。
- **stability**: 复现性单点疑虑（#183 未获回应）。

**技术备注**: Jupyter Notebook 为主；模型分 Super 64B / Nano 16B 等档；面向 Omniverse 物理仿真生态。

### [lfnovo/open-notebook](https://github.com/lfnovo/open-notebook) — NEW ⭐ 25,094（+212）| TypeScript | 信源：中

**它是什么**: Google NotebookLM 的开源自托管替代品。你把一堆资料（PDF、视频、音频）上传，它帮你做问答、摘要、甚至生成多人对话式播客；区别是数据全在你自己的服务器上，可以接本地模型（Ollama/LM Studio）离线跑，不必把资料交给 Google。

**解决什么问题**: NotebookLM 好用但数据要上传到 Google、且不可定制。open-notebook 让注重隐私或要私有部署的人，在本地用 Docker 跑一套等价能力，并能在 18+ 个模型 provider 间自由切换。

**谁在用、怎么用**（从 Issues/媒体提取真实场景）:
- 自托管/隐私优先用户：本地 Docker + SurrealDB，配 Ollama 跑离线 RAG，或接 OpenAI/Anthropic/Groq。
- 上传 PDF/视频/音频做问答、摘要、多发言人播客生成（#189 求 EasyPanel 部署模板、#257 求重复源检测，均为功能请求）。

**社区怎么评价**:
- 👍 正面: 多 provider、隐私自托管、REST API、多发言人播客是被反复称道的卖点；维护活跃（v1.9.0 发于 2026-06-02，46 contributors）。
- 👎 负面: #411"failed to send message"聊天报错（安装类痛点）；独立测评普遍指出自托管部署成本/API 配置门槛高；仍是单用户应用，不支持多用户。
- 📊 健康度: 25,094 stars / 125 watchers ≈ 200:1（严重失衡）；2,921 forks 占 stars 约 12%（多为 Docker 自部署 clone）；open issues 143 但仅 3 个评论数 >5（最高 #189 仅 12 评）→ 讨论密度极低。

**风险与争议**:
- **vanity**: stars:watchers 200:1 + HN/Reddit/X 三处皆冷，口碑高度依赖博客/榜单二次稿（XDA/KDnuggets/Medium/The New Stack）→ 媒体放大 > 真实社区沉淀。
- **stability**: 安装/配置门槛高，聊天报错类 issue 反映部署链脆弱。

**技术备注**: 标 TypeScript（Next.js 前端 1.5M 行 > Python 后端 806K），非纯 Python；后端默认 SurrealDB。

### [github/copilot-sdk](https://github.com/github/copilot-sdk) — NEW ⭐ 8,994（+38）| Java（多语言）| 信源：中

**它是什么**: GitHub 官方出的 SDK，让你把 Copilot 的"agent 能力"嵌进自己的应用和服务里。不用自己从零写一套 agent 循环（上下文管理、工具编排、模型路由、MCP、权限），直接调 SDK 就能在你的 app 里跑一个会用工具、能干活的 AI agent。6/3 刚 GA，MIT 协议，提供 TS/Python/Go/.NET/Rust/Java 多语言。

**解决什么问题**: 自建 agent loop 是重复造轮子且容易出错的活。copilot-sdk 把这套基础设施标准化，开发者只关心业务逻辑，agent 的"骨架"由 GitHub 维护。

**谁在用、怎么用**（从 Issues/X 提取真实场景）:
- 已有可用产出：github-sre-agent、RepoCheckAI、微软 copilot-sdk-samples 的 PagerDuty/Datadog 集成。
- KOL 真实 demo：Lena Hall 的 HackerNews agent（5 函数）、reddobowen 的 Flight School（Next.js）、Marko Denic 的 RSS 自动化。
- 典型场景：incident response bot、CI/CD 失败三联诊断、基础设施校验、部署助手、合规检查、企业内部 agent。

**社区怎么评价**:
- 👍 正面: 定位解决真痛点（免去自建 agent loop）；官方与高管强推（Amanda Silver、Evan Boyle）。
- 👎 负面: #558 CLI 持续 `TimeoutError: Timeout after 60s waiting for session.idle`（加超时无效）；#860 MCP server 用裸名配置时 agent 看到零个工具；#41 缺结构化输出；#73 view 工具行号无法关闭。HN 基本无讨论（官方公告帖最高 11 分 1 评）。
- 📊 健康度: 真实 subscribers 仅 86，stars:subscribers ≈ 105:1，71+ contributors，6/3 刚 GA。

**风险与争议**:
- **dependency**: 底层强绑 Copilot CLI 进程，Go/Java/Rust 需手动装 CLI。
- **cost**: premium request 配额计费（虽支持 BYOK 缓解模型层锁定）。
- **stability**: 刚 GA，核心 bug（超时 #558、MCP 工具不可见 #860）未解。
- **vendor lock-in**: 强绑 GitHub Copilot 生态；另有非官方同名 copilot-sdk（带"not supported by GitHub"声明），选型须确认官方仓。

**技术备注**: 架构 SDK → JSON-RPC → Copilot CLI（server mode）。

**风险与争议**:
