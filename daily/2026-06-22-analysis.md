### [smicallef/spiderfoot](https://github.com/smicallef/spiderfoot) — NEW ⭐ 18,781（+294）| Python | 信源：强

**它是什么**: 一个把"网络情报收集"全自动化的开源工具。你给它一个目标（一个 IP、域名、邮箱、用户名或公司名），它会自动调用 200 多个数据源去挖与这个目标相关的一切公开信息——子域名、关联 IP、泄露的邮箱密码、社交账号、开放端口、证书等，最后汇总成一张"攻击面地图"。

**解决什么问题**: 在它之前，安全人员做信息收集（OSINT）要手动一个个跑 theHarvester、Recon-ng、Maltego、各种 whois/DNS 查询，再人工拼接结果。SpiderFoot 把这些数据源编排成一次自动化扫描，跑完直接出关联图谱，省掉大量手工串联工作，且多数模块不需要 API key 即可用。

**谁在用、怎么用**（来自 Issues/HN/评测）:
- 渗透测试侦察：安全团队自托管 Docker/pip 跑本地 web UI（5001 端口），对目标做 IP/域名/邮箱/ASN footprinting。
- 攻击面映射与威胁情报：企业用它持续监控自己暴露在公网的资产。
- 社区常把它与 Maltego/theHarvester/Recon-ng 并列推荐，定位"广度采集 + 自动关联"。

**社区怎么评价**:
- 正面: 模块化（200+ 模块、多数免 API key）、采集广度受认可；HN 历史高峰 272 分（2020）。
- 负面: 稳定性与运行问题集中——#1635「Error execution」(24 评)、#1650「运行约 1 小时后冻结无报错」(21 评)、#1768「无数据 + 报错」(16 评)，且这些 issue 跨 2022–2023 长期未闭合；常见吐槽"输出数据量大需过滤""API key 配置繁琐"。
- 健康度: 18,781 stars / 402 subscribers ≈ 47:1（成熟工具属健康）；forks 3,107；open issues 269（积压明显）；约 57 贡献者（非单人）；最后 push 2026-04-13，约 2 个月无提交，维护节奏明显放缓；最新 release 仍是 v4.0，无新版本事件。

**风险与争议**:
- **bus_factor**: 商业版 SpiderFoot HX 已被 Intel 471 收购，开源版维护投入存疑，2 个月无 push + 269 积压 issue 是放缓信号。
- **stability**: 长周期运行冻结、无数据等问题长期未解。
- **privacy**: 商业 HX 云端 lock-in，敏感调查数据上云存隐私顾虑。
- **vanity**: 今日 re-trending 非新发版驱动，更像算法回流 + OSINT 话题带动，star 增量与真实事件不完全对应。

**技术备注**: Python；自托管，pip 或 Docker 部署，web UI 默认 5001 端口；GPL/MIT 双授权；v4.0 引入 correlation 规则集。

### [topoteretes/cognee](https://github.com/topoteretes/cognee) — NEW ⭐ 18,662（+347）| Python | 信源：中

**它是什么**: 给 AI agent 用的"开源长期记忆引擎"。它把异构文档（PDF、网页、代码、对话记录等）摄入后，自动构建成一张**类型化知识图谱**，让 agent 在多次会话之间记住事实、并能做跨文档的深度推理，全部可自托管。

**解决什么问题**: 没有它之前，agent 的"记忆"要么塞进 prompt（贵且有限），要么用向量库做相似检索（召回散、缺结构）。Cognee 用 ECL/GraphRAG 思路把文档变成有实体和关系的图，让"从多份文档里推理出答案"这类深度检索成为可能，定位与"即插即用聊天记忆"的 mem0 错位。

**谁在用、怎么用**（来自 Issues/评测）:
- 企业知识库、研究型 agent、代码库多文档推理——需要"结构化跨文档检索"的场景。
- 第三方评测共识："深度知识检索选 Cognee"；而 mem0 偏聊天机器人即插即用记忆、Zep 偏对话时序事实追踪。
- Issues 多为团队自身工程讨论：#2871 worker-per-task 并发执行器（40 评）、#2826 LanceDB 并发 RPC 性能、#2881 并发 embedding——痛点集中在大规模摄入性能。

**社区怎么评价**:
- 正面: 知识图谱 + GraphRAG 定位清晰，第三方对比评测认可其"深度检索"差异化；昨日仍有 push，活跃维护。
- 负面: 多次 Show HN（2024-11/2025-02/2025-06）最高仅 9 分、评论近零，热度未引爆；Reddit/X 无独立实证讨论；open issues 多由维护方驱动，缺独立用户报障。
- 健康度: 18,663 stars / 71 subscribers ≈ 263:1（比值偏高，对热门项目属结构性常见）；forks 1,969；open issues 81；2023-08 创建（近 3 年）；背后 Cognee 公司（cognee.ai）公司+开源双轨；Apache-2.0。

**风险与争议**:
- **vanity**: stars 高但 HN/Reddit 实证讨论稀薄、subscribers 极低，热度与社区足迹有落差。
- **compliance**: 无 SOC2/HIPAA 认证，受监管行业落地受限。
- **bus_factor**: 贡献者集中、issue 由维护方驱动。
- **dependency**: agent memory 赛道竞争激烈（mem0 stars 远超），定位需持续证明差异化。

**技术备注**: Python；自托管知识图谱引擎，ECL/GraphRAG；Apache-2.0；与 mem0/Zep/Letta 同赛道不同定位。

### [mikumifa/biliTickerBuy](https://github.com/mikumifa/biliTickerBuy) — NEW ⭐ 3,719（+67）| Python | 信源：强

**它是什么**: B 站会员购的抢票辅助工具。漫展、演出、谷子等限量票在 B 站会员购开售时常常秒空，这个图形化工具帮你在开售瞬间自动下单。

**解决什么问题**: 手动抢限量票几乎抢不到——填地址、选购买人、过验证码都来不及。这个工具让用户提前配好收货地址和购买人，开售瞬间自动识别滑块验证码、用 NTP 毫秒级校时卡点下单，并支持多平台通知。

**谁在用、怎么用**（来自 Issues/中文社区）:
- 漫展/演出/谷子限量票秒杀：提前配置 → 自动过滑块 + 校时 → 开售瞬间下单。
- 部署灵活：Windows 原生 / Docker / 树莓派 ARM 均可跑。
- Issues 都是真实抢票场景 bug：#963 429 光速重试触发 412 风控（最热，6 评）、#969 log file not found、#966 502、#961 重复准备订单。

**社区怎么评价**:
- 正面: CSDN/GitCode 多篇实战教程称"5 分钟上手、图形化、自动过滑块、毫秒级 NTP 校时、多平台通知"；GitHub_Daily 推荐过。
- 负面: 核心抢票路径有 412 风控、重复下单等未解 bug。
- 健康度: 3,719 stars / 14 subscribers ≈ 266:1（抢票工具类典型——大量收藏不 watch，非营销虚高）；forks 466（极高 fork/star 比，真实二次开发信号）；open issues 仅 6（活跃维护）；2023-04 创建、最近 push 2026-06-21（持续维护近 3 年）。

**风险与争议**:
- **compliance**: 抢票辅助处灰色地带，违反 B 站会员购 ToS；作者免责声明"勿盈利、后果自负、仅供学习"。
- **dependency**: 强依赖 B 站接口与风控，接口/网关一变工具随时可能失效（#963/#966 已现）。
- **stability**: 抢票核心路径存在 412 风控、重复下单等未解 bug。

**技术备注**: Python；图形化客户端，支持 Windows / Docker / 树莓派 ARM；滑块验证码自动识别 + NTP 校时。
