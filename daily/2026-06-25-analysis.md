### [interviewstreet/hiring-agent](https://github.com/interviewstreet/hiring-agent) — NEW ⭐ 2,231（+203）| Python | 信源：中

**它是什么**: HackerRank 官方组织（interviewstreet）开源的简历自动评分 agent。把候选人简历 PDF 转成结构化数据，用 LLM 抽取信息，再拉取其 GitHub profile/repos 做信号增强，按"开源贡献 / 个人项目 / 生产经验 / 技术技能"四个维度自动打分。

**解决什么问题**: 技术招聘里人工筛简历慢且主观。这个项目把 HackerRank 内部 ATS（申请人追踪系统）评分流水线开源出来，让招聘方能用候选人的真实 GitHub 实绩客观打分，也让候选人看到"简历为什么被打这个分"。

**谁在用、怎么用**（来自 Issues）:
- 招聘方/HM 自动筛选：社区已有人二次开发加 Web UI 和 Claude/OpenAI provider（todddong/hackerrank-resume-ats）。
- 扩展评估源：#175（6 评论）请求接入 LeetCode/HackerRank/CodeForces 成绩；#183（6 评论）请求加候选人可信度/真实性评估。

**社区怎么评价**:
- 正面: HackerRank 官方背书，定位清晰（开源 ATS 评分）。
- 负面: #186（7 评论，最热）反映 Gemini 免费层限流时 agent 直接卡死无法完成。
- 健康度: 2,231 star / subscribers 仅 4（≈558:1，极端失衡）；open issues 218 偏高；核心贡献者 2 人（sp2hari 48 + anxkhn-hacker 16）其余多为一次性 PR；仓龄约 11 个月仍活跃。

**风险与争议**:
- **vanity**: stars/subscribers 558:1，HN/Reddit 无有机讨论，传播靠 LinkedIn / 视频自媒体放大。
- **dependency**: 强依赖 Gemini / 外部 LLM API，限流即崩（#186）。
- **bus_factor**: 核心贡献者 2 人，其余皆一次性提交。
- **compliance**: AI 简历打分天然涉招聘偏见与候选人数据处理争议（业界有 Amazon 废弃案例）。

**技术备注**: Python 3.11+，MIT；支持 Ollama 本地 / Google Gemini，简历转 Markdown → LLM 抽 JSON → GitHub 信号增强。

**技术备注**: Kotlin，AGPL-3.0；USB + 无线（配套 Wireless Helper + mDNS/NSD）；Play Store 另有付费版（AGPL + 商业双轨）。

### [stablyai/orca](https://github.com/stablyai/orca) — NEW ⭐ 6,835（+331）| TypeScript | 信源：中

**它是什么**: 管理"一队并行 coding agent"的桌面 IDE（ADE，Agent Development Environment）。一条 prompt 可扇出多个 agent（Claude Code / Codex / OpenCode / Grok 等），每个跑在独立 git worktree，用自己的订阅账号（BYO subscription），无需在 Orca 侧登录。厂商 Stably AI 是 YC 公司。

**解决什么问题**: 同时跑多个 coding agent 时，传统终端难以并行管理、对比、择优。Orca 把每个 agent 隔离到 worktree，并提供 diff 行级标注回送 agent 修订、PR review 一等公民、账号限流切换。

**谁在用、怎么用**（来自 X / Product Hunt）:
- 并行多 agent 工作流：X（@orca_build）口碑集中在"终端分屏 + 任意 harness""PR review 一等公民"。
- worktree 隔离对比：Product Hunt 4.7（65 评），用户提出 worktree 磁盘清理（node_modules 膨胀）等实操疑问。

**社区怎么评价**:
- 正面: BYO subscription + worktree 隔离范式获 X / PH 早期好评。
- 负面: 660 open issues 多为 feature request / bug 堆积；评论最多的 issue #4994 竟是测试帖（"Test discord message"），缺高质量社区沉淀。
- 健康度: 6,835 star / subscribers 仅 15（≈456:1）；贡献者高度集中于 Stably 内部团队；仓龄仅约 3 个月。

**风险与争议**:
- **vanity**: subs 仅 15、HN/Reddit 零讨论 vs 6.8K star，传播集中厂商自有渠道（X / Discord / PH）。
- **bus_factor**: 贡献者高度集中于厂商内部，外部贡献尾部零星。
- **dependency**: 依赖外部各家 agent CLI 与订阅可达性。

**技术备注**: TypeScript，MIT；桌面（macOS/Windows/Linux）+ 移动伴侣 app；官网 onorca.dev。

### [kunchenguid/no-mistakes](https://github.com/kunchenguid/no-mistakes) — NEW ⭐ 2,187（+110）| Go | 信源：中

**它是什么**: 一个本地 git proxy / 校验门禁。当你 push 到 no-mistakes remote 后，它在一次性 worktree 里跑一条固定 pipeline（intent → rebase → review → test → document → lint → push → pr → ci），全过校验通过才转发到 upstream 并自动开 PR、看护 CI。（repo description 即 `git push no-mistakes`。）

**谁在用、怎么用**（来自 Issues / HN）:
- 给 agent 大 diff 做 push 前门禁：agent-agnostic（claude / codex / opencode 等），提供 TUI + `/no-mistakes` agent skill。
- 真实使用调试：#255 CI 步骤在 daemon 非 repo 目录下挂死、#287 Windows 下 Codex 子进程闪窗、#320 worktree git hook 路径错乱——多个跨平台 / CI 集成 bug 说明确有人真跑。

**社区怎么评价**:
- 正面: 定位切中"agent 大 diff 审查瓶颈"，#305 有深度使用者提改进建议。
- 负面: HN "Show HN: Git Push No-Mistakes" 仅 19 分 / 6 评论，热度偏低；跨平台 / 路径 / CI 集成稳定性问题集中。
- 健康度: 2,187 star / subscribers 仅 4（≈547:1）；实质单作者（kunchenguid 282 commits + bot 73，其余 3 人各 1-2 次）；仓龄仅约 2.5 个月。

**风险与争议**:
- **vanity**: 547:1 比值 + HN 低分 + 无 X/Reddit 足迹，热度远超社区足迹。
- **bus_factor**: 实质单作者。

**技术备注**: Go，MIT；agent-agnostic，TUI + `/no-mistakes` agent skill；本地 git proxy 转发 upstream。

## 如何理解今天的热度信号

今日榜单有两类"高 stars/subscribers 比值"，读法不同：

- **短龄厂商主导 agent 项目**（hiring-agent 558:1、no-mistakes 547:1、orca 456:1、design.md 163:1）：仓龄 2.5~3 个月，传播集中在厂商自有渠道（X / PH / LinkedIn / 官方背书），HN/Reddit 缺独立讨论——这类比值 + 短龄 + 单一传播路径的组合，是"关注度爆发但社区足迹薄"的信号，需谨慎区分热度与采用。
- **真实刚需但社区隔离的项目**（zapret 91:1、headunit-revived 68:1）：比值也偏高，但 issue 里有大量真实使用调试痕迹（zapret #5704 720 条俄语求助、headunit #219 60 条硬件调试），高比值更多源于语言 / 地缘 / 垂直硬件圈子隔离，而非空心化。

结论：stars/subscribers 比值不能单独定性 vanity，必须结合仓龄、issue 真实性、传播路径综合判断。apple/container 与 flutter 这类超大成熟项目的高比值则是结构性的，不适用同一标准。
