## NEW 项目分析

### [allenai/olmocr](https://github.com/allenai/olmocr) — NEW ⭐ 18,293（+334）| Python | 信源：强

**它是什么**: Allen Institute for AI（Ai2，知名非营利 AI 研究机构）出的一个 PDF/文档「解版式」工具。它把 PDF 或扫描图像里的文字，按正确的阅读顺序抽成干净的纯文本，专门用来给大模型做训练语料。

**解决什么问题**: 用 PDF 做 LLM 训练数据时，直接抽文本会得到乱序、混着页眉页脚公式的脏内容。olmocr 用一个 7B 视觉语言模型，结合文档里嵌入的文本和空间位置信息，输出结构化、按阅读顺序排好的文本，且成本低（约 $190/百万页，宣称约 GPT-4o 的 1/32）。

**谁在用、怎么用**（来自 Issues/HN/X）:
- 真实部署与微调需求集中：#120 训练流程（23 评）、#33 macOS 支持（28 评）、#111 ROCm/AMD 支持（11 评）、#302 B200 性能（15 评）——高互动来自"想跑起来"而非报错抱怨。
- 二次开发采用真实存在：Reducto AI 用 olmOCR 数据集训练出 RolmOCR（Apache 2.0）并公开。
- 非英文场景暴露质量问题：#184 中文图片输出 jsonl 乱码（11 评）。

**社区怎么评价**:
- 正面: 首发 Show HN 313 分/40 评、关联 OCR Arena 216 分/63 评；KOL（Rohan Paul、Charly Wargnier）转发放大低成本卖点；被拿来与 Marker、Docling 对比。
- 负面: 默认要 NVIDIA GPU，macOS/AMD 长期无原生支持；中文等 CJK 输出质量有报告问题。
- 健康度: 18,293 ⭐ / 100 subscribers（183:1，大机构大 fork 结构性正常）、forks 1,506、15 名机构团队贡献者（非单人）、open issues 82、创建于 2024-09，属成熟机构项目本次首入 tracker。

**风险与争议**:
- **dependency**: 默认依赖 NVIDIA GPU + vLLM，macOS 与 AMD ROCm 支持缺失（#33/#111）。
- **stability**: 非英文（中文）输出质量有报告问题（#184）。

**技术备注**: Python，基于 7B 视觉语言模型；最新 olmOCR 2 得分 82.4，Apache 2.0 生态友好。

### [TencentCloud/CubeSandbox](https://github.com/TencentCloud/CubeSandbox) — NEW ⭐ 6,814（+79）| Rust | 信源：中

**它是什么**: 腾讯云出的一个给 AI agent 用的「沙箱」——让 AI 生成的代码可以在一个隔离、安全、启动极快的环境里跑，不会影响宿主机。定位是开源版的 E2B 替代。

**解决什么问题**: AI agent 要执行不可信代码（跑用户脚本、编译、联网）时，直接在服务器上跑有安全风险，用 Docker 又因共享内核隔离不彻底。CubeSandbox 走 microVM 路线（RustVMM+KVM + eBPF 网络隔离），既有虚拟机级隔离又快，宣称冷启 <60ms、单实例内存 <5MB、96 核可跑 2000+ 沙箱，且改一个 URL 环境变量就能从 E2B 迁移过来。

**谁在用、怎么用**（来自 Issues/媒体）:
- 官方声称内部已处理数百亿请求、外部 MiniMax 用于训练场景——但均为厂商自述。
- 最热 issue #644（111 评）实为官方「犀牛鸟」悬赏任务（集成 Claude Code/CodeBuddy/OpenCode），是运营活动而非自发用户反馈。
- 0.4.0 新增 CubeEgress（58 commits/15 贡献者），维护活跃。

**社区怎么评价**:
- 正面: 对标 Firecracker microVM 路线，E2B drop-in 兼容降低迁移成本，技术媒体（aibase/apidog/ZenML）有报道。
- 负面: HN 6 条提交几乎零讨论（最高 5 分 0 评、Show HN 7 分 2 评）、X 仅官方与媒体转述、Reddit 仅在 E2B alternatives 综述中列名。
- 健康度: 6,814 ⭐ / 28 subscribers（243:1，大厂官方项目常态）、forks 569、10 贡献者但 fslongjin 152 commits 绝对主导、创建约 3 个月、Apache 2.0（Rust 52%+Go 26%+C 19%）。

**风险与争议**:
- **vanity**: stars 领先但 HN/X/Reddit 独立讨论近乎为零，最热 issue 是官方悬赏而非有机反馈。
- **bus_factor**: 贡献高度集中于单个核心作者。
- **dependency**: 宣称的冷启/密度/内存基准全部来自厂商自述，无第三方验证，需自测。
- **stability**: 仓龄仅 3 个月，长期稳定性待观察。

**技术备注**: Rust，RustVMM+KVM microVM + eBPF，Apache 2.0，E2B 兼容 API。

### [togatoga/karukan](https://github.com/togatoga/karukan) — NEW ⭐ 588（+42）| Rust | 信源：中

**它是什么**: 一个跑在 Linux 和 macOS 上的日语输入法（IME），用神经网络做假名到汉字的转换。受 azooKey 启发，底层用 llama.cpp 跑一个 GPT-2 模型来做上下文感知的转换和用户选词学习。

**解决什么问题**: 传统日语输入法的假名汉字转换靠统计词典，对长句、上下文相关的转换不够聪明。karukan 用神经网络做实时转换（live conversion），转换质量更贴合语境，且完全本地运行。

**谁在用、怎么用**（来自 Issues/日语技术社区）:
- 真实使用中的输入模式/崩溃反馈：#20「Fcitx 5 が落ちる」（fcitx5 崩溃，4 评，最热）、#37/#33 macOS US 键盘布局下 Shift 英字输入后无法回到平假名模式（各 2 评），#38 是针对 #37 的修复 PR。
- #21 请求 Flatpak 打包 + Release 流程，显示有用户在实际尝试安装分发。
- 日语社区（Qiita/Zenn）有独立技术评测（「KarukanというLinux向け日本語入力システムがすごい」），已催生 Windows 衍生项目 rakukan。

**社区怎么评价**:
- 正面: 日语开发者社区独立评测认可其神经转换效果；已有衍生项目。
- 负面: fcitx5 崩溃、macOS 模式回退等基础输入 bug；安装门槛高（需 llama.cpp + fcitx5 + Rust 工具链）。
- 健康度: 588 ⭐ / 5 subscribers（118:1，绝对量小）、forks 35、open issues 12、贡献者仅作者 togatoga 1 人（26 commits）、创建 2024-08、2026-06-24 仍活跃。

**风险与争议**:
- **bus_factor**: 单人维护，贡献高度集中。
- **stability**: fcitx5 崩溃、macOS 模式回退等基础输入 bug，未达生产级。
- **compliance**: 引擎数据 derived from Mozc（BSD-3），需注意数据授权边界。
- **dependency**: 依赖 llama.cpp + fcitx5 + Rust 工具链，安装门槛高。

**技术备注**: Rust，llama.cpp 跑 GPT-2，Linux(fcitx5)/macOS(InputMethodKit)；英语圈 grounding 弱，X 仅作者自宣传，HN/Reddit 无讨论。
