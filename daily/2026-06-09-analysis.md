### [google/skills](https://github.com/google/skills) — NEW ⭐ 12,458（+461）| Python | 信源：中

**它是什么**: Google 官方的"Agent Skills 仓库"——把 Google Cloud 各产品（BigQuery、GKE、Cloud Run、Cloud SQL、Firebase、Gemini API 等）的专家知识压缩成一个个 skill 包，让 AI agent 在需要时按需加载，而不是把整套文档塞进上下文。Cloud Next 2026 Day 1 发布。

**解决什么问题**: 之前让 AI agent 用某个云产品，要么靠 MCP 把产品的全部接口/文档返回给模型（context 爆炸、噪声大），要么靠模型自己记忆（容易过时编造）。skills 的思路是"按需取用"——agent 判断这次任务需要 BigQuery 知识，才加载 BigQuery skill。官方还附了 Well-Architected（安全/可靠/成本）三支柱 skill 和 Onboarding/Auth/Networking 等 recipe。

**谁在用、怎么用**（来自 Issues/官方渠道）:
- 场景 1: 给 Antigravity / Gemini CLI / 第三方 agent 装上 Google Cloud 产品知识，`npx skills add google/skills` 安装后让 agent 在写部署/查询脚本时调用。
- 场景 2: issue 区全是"希望覆盖更多 Google 产品"的功能请求——#46 请求 Google Maps skill、#104 Spanner、#73 Google Ads、#58 PII 脱敏、#48 versioning 诉求，侧面印证真实采用意愿。

**社区怎么评价**:
- 正面: DEV 社区有深度分析（"MCP = 返回一切，Skills = 按需取用"），forks 970 远超 subscribers，说明真有人 clone 来用/改。
- 负面: Google 自己也承认 skill 效果不稳定——Gemini 3 基线下有时 AGENTS.md 直接指令比 skill 更管用。
- 健康度: 12,458 stars / 87 subscribers（143:1，官方大厂仓结构性偏高，不单凭此判 vanity）；open issues 仅 12，Apache-2.0，活跃维护；但贡献者实质只有 Google 内部 `cloud-ix-copybara` 同步机器人（38/42 提交）——这是 Google 内部 monorepo 的单向镜像仓，外部协作通道很弱。

**风险与争议**:
- **vanity**: 曝光由官方投放驱动（Cloud Next 发布 + @GoogleCloudTech 推广），HN（"Google Unveils Agent Skills Repository" 仅 5 分 1 评论）和 Reddit 自发热度都很弱，传播路径与社区 grounding 存在落差。
- **bus_factor**: 实为 Copybara 镜像仓，外部 PR 几乎无法合入；但有官方背书，维护中断风险有限。

**技术备注**: Python；Apache-2.0；`npx skills add google/skills` 安装；覆盖 GCP 主要产品 + Well-Architected 三支柱。

### [Andyyyy64/whichllm](https://github.com/Andyyyy64/whichllm) — NEW ⭐ 3,506（+143）| Python | 信源：中

**它是什么**: 一个本地 LLM 选型 CLI——自动检测你机器的 GPU/CPU/RAM，再用真实跑分（LiveBench、Artificial Analysis、Aider、Chatbot Arena ELO 等 recency-aware 榜单）从 HuggingFace 里排出"你这台机器能跑、且性能最好"的本地模型，一条命令直接下载运行。强调"按真实跑分排，不按参数量排"。

**解决什么问题**: 之前想在本地跑 LLM，要自己估算显存够不够、量化到几 bit、哪个模型在自己硬件上又能跑又不傻——全靠经验和试错。whichllm 把硬件检测 + 量化惩罚 + partial offload + MoE 处理 + 真实跑分合在一起，给出"能跑且最优"的推荐，还支持 `whichllm run` 一键下载对话、`--gpu` 模拟换硬件的购买决策、Ollama 集成。

**谁在用、怎么用**（来自 Issues）:
- 场景 1: 本地 LLM 玩家装上后跑一条命令拿到适配自己显卡的模型推荐 + 直接运行。
- 场景 2: 外部贡献者持续补硬件目录——#68 修 AMD Linux 独显检测、#47 扩 GPU catalog、#65 请求多 GPU 异构模拟（`--gpu` flag）。

**社区怎么评价**:
- 正面: 作者响应快、迭代密（v0.5.8），有真实外部 PR（devangpratap、akx、larsblumberg、max-wittig 等），不是空壳。
- 负面: 最热 issue **#76**（15 评论）"A3000 6GB 推荐很差"——给 6GB 笔记本推 Q8 模型明显不合理，作者当场承认"recommendation this far off… does look wrong"，暴露排序逻辑在小显存边缘场景失准；#85 抓取模型报错。
- 健康度: 3,506 stars / 14 subscribers（251:1，榜单最极端比值）；25 open issues；贡献者 10+，核心作者 175 commits。

**风险与争议**:
- **vanity**: 251:1 比值 + HN（Show HN 仅 3 分 0 评论）/ Reddit / X 三处近乎零有机讨论，热度与社区足迹不匹配，需谨慎解读为"真实采用"。但与纯营销样本不同，它有真实工程活动支撑——更像"被榜单推上去吸了大量观望 star"。
- **stability**: benchmark 合并/normalize 逻辑在低显存场景给出明显错误推荐（#76 实证，作者承认），方法论尚不稳健。

**技术备注**: Python；MIT；检测硬件 + recency-aware 跑分排序；支持 Ollama 集成 + JSON 输出 + `--gpu` 硬件模拟。

## 如何理解今天的热度信号

今天最值得读者带走的一条方法是：**高 stars/subscribers 比值今天必须分两类看**。

榜单上比值超过 150:1 的项目有近一半，但它们的"质地"完全不同：

- **真项目被榜单放大型**（whichllm 251:1、CopilotKit 199:1、goose 197:1、mempalace 174:1）：比值高，但 issue 区有真实工程讨论、外部贡献者提 PR、版本持续迭代。高比值主要来自"上榜后吸引大量观望 star，而 watch 这种重度行为跟不上"，不等于空心。
- **传播驱动型**（career-ops 244:1、Agent-Reach 315:1）：比值高 + 仓龄短 + HN/Reddit/X 三处零有机讨论 + 外部声量集中在作者可控渠道（ProductHunt / 自营博客 / KOL newsletter）。这类才需要把"Trending 热度"和"真实采用"明确切开。

判断一个短龄高热项目是否需要谨慎解读，看的是**「高比值 + 零社区足迹 + 仓龄短 + 营销集中」的组合**，而不是比值这一个数字。同样，google/skills、openai/plugins 这类官方仓的高比值是结构性的（大厂仓 watch 行为本就稀缺），不应与刷榜混为一谈——它们的真正看点是"官方按需知识包对抗 MCP context bloat"这个技术方向，而非热度本身。
