## 首次上榜项目

### [music-assistant/server](https://github.com/music-assistant/server) — NEW ⭐ 1,850（+20）| Python | 信源：强

**它是什么**: 一个自己部署的"家庭音乐总调度台"。它把你订阅的流媒体（Spotify、Apple Music 等）和家里五花八门的音箱（Sonos、AirPlay、DLNA、ESPHome 智能音箱）连起来，让你在所有房间同步播放、统一控制。最常见的形态是作为智能家居平台 Home Assistant 的一个组件运行。

**解决什么问题**: 没有它之前，不同品牌音箱各用各的 App，Spotify 推一套、AirPlay 推一套，多房间同步要么买同品牌全家桶要么放弃。装上之后，一个面板就能把任意流媒体源推到家里任意音箱，并接入语音/自动化（比如"回家自动在客厅放歌单"）。

**谁在用、怎么用**（来自 Issues / HA 论坛）:
- 场景 1：跑在 Raspberry Pi / NAS / Intel NUC 等常开设备上，作为 Home Assistant Add-on，把 Spotify 桥接到全屋音箱做多房间同步播放（强依赖 mDNS/uPnP，必须同一二层网络）。
- 场景 2：呼声最高的需求是 PulseAudio 支持（#3724，52 评论）和扩展更多流媒体 provider（Pocket Casts / Storytel，#3127/#4054）。

**社区怎么评价**:
- 正面: 2.8 修复 snapcast TTS 同步、新增 party mode + 访客 QR 码；2.9.0 的自研开源协议 Sendspin 扩展到 ESPHome/LedFx/ESP32 第三方客户端 + Philips Hue 随节拍同步，被认为是协议外溢到更大生态的里程碑。
- 负面: 2.8 把 AirPlay/DLNA/MusicCast/Sendspin 合并成单一 player 实体并删掉其余，破坏了依赖特定协议的自动化（Yamaha 功放用户要求能否切回独立 player），后来在高级设置找到 workaround。
- 健康度: stars/subs ≈ 92:1 看似偏高，但自托管/HA 生态用户聚在官方论坛和 Discord、不在 GitHub watch，属结构性；30 贡献者、82 open issues、52 PR 在审、6/13 当天仍有 commit，三月一版的稳定节奏，维护活跃。

**风险与争议**:
- **stability**: 2.8/2.9 的多协议 player 合并是破坏性变更，已造成既有自动化中断，升级需谨慎。

**技术备注**: Python；2019 年起的成熟自托管项目，本次因 2.9.0（6/10）发版 + Sendspin 协议跨生态落地上榜。无 vanity/compliance 隐患。

### [iptv-org/iptv](https://github.com/iptv-org/iptv) — NEW ⭐ 118,160（+179）| TypeScript | 信源：强（知名老项目首入 tracker）

**它是什么**: 一个收集全球公开 IPTV 直播频道地址的超大社区清单仓库。任何人都能拿这份 M3U 播放列表导入播放器看世界各地的公开电视频道。

**解决什么问题**: 公开 IPTV 频道地址分散、失效快、难维护。这个仓库用机器人每天自动更新播放列表和流地址（6/13 仍是 `[Bot] Update PLAYLISTS.md` / `Update /streams` 在跑），社区持续提交新频道（#38879 加 4ever Cinema、#38806 加 ETNO Kanal），等于一份持续维护的全球频道目录。

**社区怎么评价 / 为何上榜**: 118K stars、2,165 subscribers（比值健康 ~55:1）的长青社区仓，周期性回到 trending，通常由频道批量更新和外部转载带动。这是知名老项目首入本 tracker，按轻量条目记录，不做完整 grounding。

**风险与争议**:
- **compliance**: IPTV 频道地址的版权/落地合规因地区而异，仓库本身仅聚合公开地址，使用者需自行评估当地合规性。

**技术备注**: TypeScript（脚本与自动化）+ 自动化 bot 维护 + M3U 播放列表。

### [LMCache/LMCache](https://github.com/LMCache/LMCache) — NEW ⭐ 8,695（+28）| Python | 信源：强

**它是什么**: 给大模型推理加的一层"KV 缓存复用层"。大模型回答问题时会算出大量中间状态（KV cache），通常一次性用完就丢；LMCache 把这些状态存下来（内存/Redis/S3 等多后端），下次遇到相同前缀直接复用，省掉重复计算。

**解决什么问题**: 没有它之前，同一段系统提示或长文档每次请求都要重新做 prefill，首字延迟（TTFT）高、GPU 算力浪费。用了之后做到 prefill-once / reuse-everywhere，官方称 TTFT 可降 3-10x，并支持 prefill-decode 分离部署。它是底层库，用户一般通过 vLLM / NVIDIA Dynamo 等上层间接消费，不直接装。

**谁在用、怎么用**（来自 Issues / 厂商发布）:
- 场景 1：作为推理栈的 KV cache 标准层接入——已被 **NVIDIA Dynamo 1.0（深度三层集成）、Red Hat llm-d、字节 AIBrix、KServe、vLLM production-stack** 采纳，靠 vLLM KV connector API 接入，生产级 K8s/Helm 部署成熟。
- 场景 2：open issues 全是真实集成方的深度工程协作——修 MLA/DeepSeek V3/GLM-5 多组 KV 缓存 bug（#2951，24 评论）、多 LMCache server 部署（#3248）、Rust libblkio I/O 后端（#3154）。

**社区怎么评价**:
- 正面: HN "Lossless LLM 3x Throughput by LMCache" 154 分 53 评论；多厂商生产采纳 + arXiv 企业级论文（2510.09665）；issues 无空洞，全是集成方提的功能/缺陷。
- 负面: CPU-GPU 传输每次取存有 10-50ms 延迟权衡，需按场景评估。
- 健康度: stars/subs ≈ 193:1 单看偏高，但作为底层库被间接消费、用户不直接 watch，属结构性；309 open issues 是活跃技术讨论非堆积，6/13 仍在 push，主力 ~10 人背靠 UChicago LMCache Lab + 多企业贡献者。

**风险与争议**:
- **bus_factor**: 学术 lab 主导、贡献者集中度偏高（中等风险），但已有多企业贡献者分摊。

**技术备注**: Python；2024 年创建；vLLM 生态核心组件，KV cache offloading + 多后端（CPU/Redis/S3/NIXL）。这是真实采用而非热度虚高的典型样本。
