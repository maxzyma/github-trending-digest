### [alexzhang13/rlm](https://github.com/alexzhang13/rlm) — NEW ⭐ 4,937（+43）| Python | 信源：强

**它是什么**: Recursive Language Models（递归语言模型）的官方推理库。它换了一种处理超长 prompt 的思路：不把长文本一股脑塞进模型上下文窗口，而是把它当成 REPL（编程环境）里的一个**变量**，让模型自己写代码去拆分、检索、并递归调用自身或子模型来逐步消化。

**解决什么问题**: 现在的大模型有"上下文腐烂"（context rot）问题——上下文越长，中间内容越容易被忽略，效果反而下降。RLM 让模型像程序员处理大文件那样，按需读取、分块、递归处理，从而在近乎无限长的上下文上保持效果。核心数据：GPT-5-mini 跑 RLM 在最难的长上下文基准 OOLONG 上正确数 >2× 直接用 GPT-5，且更便宜；论文还后训练了原生递归模型 RLM-Qwen3-8B。库支持多种 sandbox（含 DockerREPL）。

**谁在用、怎么用**:
- 研究/复现场景：forks 高达 824（fork/star 约 17%），大量研究者在 fork 实验、复现长上下文基准。
- 机构采用：Prime Intellect 公开将其称为"context folding 最简单灵活的方法"并作为研究重点。

**社区怎么评价**:
- 正面: HN 多个高分帖——"Recursive Language Models" 161 分 / 23 评、原始博客帖 135 分 / 35 评，讨论集中在实用性与效率，整体正面。
- 负面: 已有跟进论文（arXiv 2603.02615 等）指出"递归本身可能不是主要增益来源，多数提升来自程序化的上下文交互而非递归本身"，核心机制有效性存在学术争议。
- 健康度: 4,937 stars / 49 subscribers ≈ 100:1（热门研究项目正常区间），contributors 21 但作者占 112 提交，单核偏重；2025-12 创建仍活跃。

**风险与争议**:
- **bus_factor**: 提交高度集中于作者本人。
- **cost**: 依赖 GPT-5 / 子 LLM 多次递归调用，token 成本与 API 依赖较重。
- **dependency**: 效果绑定特定前沿模型能力，迁移到弱模型收益不确定。

**技术备注**: Python，支持多 sandbox（DockerREPL 等）；arXiv 2512.24601，配套博客与原生递归模型 RLM-Qwen3-8B。

### [DeusData/codebase-memory-mcp](https://github.com/DeusData/codebase-memory-mcp) — NEW ⭐ 5,409（+371）| C | 信源：中

**它是什么**: 一个高性能的代码智能 MCP 服务器。它把整个代码库索引成一张持久化的"知识图谱"，让 AI 编码助手（Claude Code / Cursor / Codex CLI 等）不用逐文件 grep，而是直接查调用链、影响分析、模块 hub。单个静态二进制、零依赖，号称平均仓库毫秒级索引、158 种语言、亚毫秒查询。

**解决什么问题**: AI 编码 agent 理解大型代码库时，通常靠反复 grep / 读文件填上下文，既慢又烧 token。这个项目把代码结构预先索引成图，agent 一次查询就能拿到结构化的依赖关系，号称 token 消耗减少 10x（论文）到 120x（项目页）。

**谁在用、怎么用**:
- 作为 AI 编码 agent 的结构化检索后端，提供 14 个 MCP 工具做调用链 / 影响分析 / hub 检测。
- 真实诉求样本：#351 用户主动要求支持 Git worktree 工作流。

**社区怎么评价**:
- 正面: 有 arXiv 预印本（2603.27277）背书，工具目录（PulseMCP/lobehub 等）收录。
- 负面: #333 索引静默失效——72k LOC Rust 项目返回 `status:"indexed"` 却只生成约 500 节点（预期约 5,000）；#394 Windows 平台 8 个 bug 伞状追踪。索引正确性与跨平台稳定性是真实痛点。
- 健康度: 5,409 stars / 29 subscribers ≈ 187:1（偏高）；作者 733 提交，第二名仅 32，单人主导；2026-02 创建仅 4 个月。

**风险与争议**:
- **bus_factor**: 单人主导，巴士因子≈1。
- **vanity**: 187:1 比值偏高 + HN（作者自荐贴仅 4 分）/X/Reddit 几乎零有机讨论，传播主要靠工具目录收录与自荐。
- **stability**: 索引正确性（#333）与 Windows 平台（#394）存在实质缺陷。
- **dependency**: 论文 66 语言/10x 与项目页 158 语言/120x 数字不一致，营销数字明显领先实测。

**技术备注**: C 语言，MIT，单静态二进制零依赖；arXiv 2603.27277（未经同行评审）。

## 如何理解今天的热度信号

今日两类信号需要分开看：

1. **skills 仓的超大 star 池是生态真实现象，但不等于同规模使用。** superpowers（231k）、mattpocock/skills（133k）能在 4~8 个月累积超大 star，是 Claude Code 生态"开发者收藏式 star"的产物。它们确实代表 skills 方法论的高关注度，但 subscribers/issues 远低于 star 量级，读者应理解为"被广泛收藏的配方集"，而非有同等用户基数的工程项目。

2. **高 ratio + 弱 grounding 的短龄新仓需谨慎。** Agent-Reach（366:1）、OpenMontage（105:1 但 X 纯营销 + 6 周停更）、codebase-memory-mcp（187:1 + 营销数字不一致）三者，stars 增速与 HN/Reddit/X 社区足迹存在明显落差。相比之下 rlm（论文 + HN 多帖 + 机构采用）与 iroh（1.0 发版 + 真采用）是今日 grounding 最扎实的样本，值得优先点开。
