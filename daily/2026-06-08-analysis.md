### [microsoft/pg_durable](https://github.com/microsoft/pg_durable) — NEW ⭐ 1,490（+316）| Rust | 信源：中

**它是什么**: 一个 PostgreSQL 扩展，让你把"长时间运行、不能中途丢状态的工作流"直接跑在数据库里。比如"调用外部 API → 等 3 天 → 发邮件"这种跨越长时间、必须保证每一步都不重复不丢失的任务。

**解决什么问题**: 传统做法要单独部署一套编排系统（Temporal、Airflow 等）来记录"任务跑到哪一步了、崩溃后从哪恢复"。pg_durable 把这套状态机直接放进你已有的 Postgres——工作流状态和业务数据同库，借数据库自带的时间点恢复（PITR）一起备份恢复，省掉一整套外部基础设施。

**谁在用、怎么用**（来自 Issues/HN）:
- ETL / 数据本地化场景：状态与数据同库，崩溃后用 PITR 统一恢复（HN 正面观点）。
- 但真实 durable 负载（如限流 LLM 的 429 重试）目前必须手撸 `df.loop`+`df.if`+`df.sleep`——issue #155 指出 DSL 缺声明式 retry/backoff/timeout 组合子。

**社区怎么评价**:
- 正面: HN 467 分 / 106 评论（2026-06-05），认可"DB 快照连同任务一起恢复、减少外部组件"。
- 负面: HN 评论尖锐——"不能单元测试、不能版本化"、SQL DSL 难读难维护、把扩展压力推给数据库、业务逻辑藏进 DB 形成"hidden brain"、可观测性弱于外部编排器；并质疑相对 Temporal/DBOS 的优势。
- 健康度: stars 1,490 / watchers 仅 4（~372:1）；contributors 9 人高度集中（pinodeca 124 commits），Copilot 代写 26 commits、#209 PR 标注"100% written by Claude"；Meta Bug #58 列多项已知缺陷未清。

**风险与争议**:
- **vanity**: watchers=4 与 1,490 stars 极度悬殊，热度由 HN 单点 + 微软品牌驱动，X/Reddit 零讨论。
- **stability**: 4 个月仓龄的早期实验，DSL 缺核心 resilience 能力，Meta Bug 未清，非 production grade。
- **bus_factor**: 提交高度集中于少数维护者 + AI 代写占比显著。

**技术备注**: Rust（pgrx 框架）实现的 PostgreSQL 扩展，确为 microsoft 官方 org。

### [refactoringhq/tolaria](https://github.com/refactoringhq/tolaria) — NEW ⭐ 12,951（+245）| TypeScript | 信源：中

**它是什么**: 一个桌面笔记/知识库应用，把笔记以本地 Markdown 文件存储、用 git 管理版本，并能让 AI（Claude Code / Codex）直接读写你的知识库。可以理解为"开源版 Obsidian + 原生 git 历史 + AI 接入"。

**解决什么问题**: Obsidian 强大但闭源、Notion 数据在云端。tolaria 主打本地优先（offline-first）、文件化存储、git 作为一等公民——笔记是你磁盘上的纯文本，版本历史走 git，还能把整个知识库作为 AI agent 的上下文共享。

**谁在用、怎么用**（来自 HN/评测）:
- 个人 second brain；把公司文档整理成 AI 可读的 context；让 coding agent 与人共享同一知识库。
- HN Show HN（2026-04-23）318 分 / 143 评论，定位"offline-first、file-based、对笔记组织有强观点"。

**社区怎么评价**:
- 正面: 真开源（AGPL-3.0，对比 Obsidian 闭源）、block 块编辑、git 原生版本历史、支持 MCP。媒体评测把它与 Obsidian/Notion 正面对比。
- 负面: Windows 移植质量明显弱于 macOS——#828 语言设置不持久化致"实际不可用"、#827 Windows 主菜单不可用、#826 Windows Pi CLI 启动失败；且无实时 sync，仅靠 git 同步。
- 健康度: stars 12,951 / watchers 35（~370:1）；仓龄仅约 4 个月、16 天破万星；contributors 10 人但作者 LucaRonin 一人 2,757 commits（bus_factor 集中）。

**风险与争议**:
- **vanity**: 370:1 stars/watchers + 16 天破万星 + 大量 KOL/"Karpathy LLM wiki 概念落地"式营销驱动，独立用户长期使用证据少。
- **stability**: young repo，Windows 多处不可用。
- **bus_factor**: 单人主导提交。

**技术备注**: TypeScript 桌面应用，AGPL-3.0（license 干净），支持 MCP 接入。

### [RyanCodrai/turbovec](https://github.com/RyanCodrai/turbovec) — NEW ⭐ 7,328（+1,554）| Python | 信源：弱

**它是什么**: 一个向量检索索引库（Rust 内核 + Python 绑定），主打在内存受限/ARM 环境下用极小内存做大规模向量近邻搜索。宣称 1000 万向量从 31GB 压到 4GB、零训练、比 FAISS 快。

**解决什么问题**: RAG/语义检索需要把大量向量装进内存做相似度匹配，内存是瓶颈。turbovec 用 Google 的 TurboQuant 量化算法压缩向量，号称在本地/边缘设备上也能跑大规模私密 RAG。

**谁在用、怎么用**（来自 Issues）:
- Issues 多为功能提议/工程讨论而非真实使用反馈：#68 磁盘缓存 centroid、#65 增删 benchmark、#70 公开 I/O API、#86 Swift binding、#85 Node.js binding。无独立用户晒落地案例。

**社区怎么评价**:
- HN: 仅 1 条 self-post（4 分 0 评论），等同无讨论。
- X/Twitter: 全是单向营销转发（复述 README 卖点），无独立开发者实测；且 X 上 TurboQuant 热度主要来自 Google Research 原算法，被 turbovec 蹭流量。
- Reddit: 无讨论。
- 健康度: stars 7,328 / watchers 39（~188:1）；仓龄约 2.5 月、近 9 天未更新；contributors 仅 3 人（RyanCodrai 141 commits，另两人各 1 次）。

**风险与争议**:
- **algorithm**: 核心卖点依赖的 TurboQuant 量化算法存在学术争议——RaBitQ 作者发表对称复现研究，指 Google 论文用非对称硬件设置且无法复现；对称基准下 RaBitQ 普遍持平或胜出，Milvus 选 RaBitQ、Qdrant 取混合。即核心性能主张未被独立证实。
- **vanity**: 188:1 stars/watchers，外部传播全为营销转发，HN/Reddit 零 grounding。
- **bus_factor**: 单人项目。

**技术备注**: Rust 内核 + Python bindings，底层为 TurboQuant 量化。
