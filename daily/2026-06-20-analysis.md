### [tursodatabase/turso](https://github.com/tursodatabase/turso) — NEW ⭐ 20,060（+774）| Rust | 信源：强

**它是什么**: 一个嵌入式 SQL 数据库，用 Rust 从零重写了 SQLite，并保持与 SQLite 兼容。可以理解为「下一代 SQLite」——同样是嵌进你程序里直接读写的单文件数据库，但底层换成了 Rust + 异步 IO，并原生加了向量搜索、全文搜索、嵌入式副本（本地直读 + 远端同步）这些 SQLite 本体没有的能力。

**解决什么问题**: 传统 SQLite 是 C 写的、单线程为主、没有原生向量/并发写。turso 想在保持「零运维、嵌入式」体验的同时，补上现代应用要的并发写、向量检索（给 AI/RAG 用）、以及「每个租户一个库」可扩到百万级库的架构——既能本地优先（边缘/离线读），又能和云端同步。

**谁在用、怎么用**（来自 Issues/HN）:
- 场景 1（多租户 SaaS）: database-per-tenant 模式，每个客户一个库，可扩展到百万级库，HN「Deep dive into Turso」帖（192 分/145 评论）有大量讨论。
- 场景 2（AI/RAG）: 原生向量搜索 + AgentFS（AI agent 隔离工作区），用作 agent 的本地隔离存储。
- 场景 3（并发打磨中）: #7500（Mvcc sync）、#7488 等围绕 MVCC 并发写的 issue，0.5.0 已带并发写 beta。

**社区怎么评价**:
- 正面: Rust 重写 + io_uring async、原生向量/全文搜索、嵌入式副本被认为是真正解决 SQLite 痛点；3000+ commits、50+ 贡献者，开发极活跃。
- 负面: #697（27 评论）WebContainers 加载失败；Rust 生态缺好用 ORM；HN 评论区夹带对官方曾发生客户数据丢失事件的担忧。
- 健康度: stars/subscribers ≈ 242:1（成熟基础设施结构性高比，不构成 vanity）；open issues 745（量大但今日仍在 push，活跃）；contributors 100+，仓龄约 2.8 年。

**风险与争议**:
- **stability**: 官方明确标 BETA，不建议存生产数据；HN 提及曾有客户数据丢失事件。
- **dependency**: 核心价值与 Turso Cloud 商业服务深度绑定。
- **bus_factor**: 贡献者基础尚可，但叙事高度绑定创始人（Glauber Costa / Pekka Enberg）个人推广。

**技术备注**: Rust，从 libSQL fork 演进到全量重写；io_uring 异步 IO，Tantivy 全文搜索，SQLite wire 兼容。

### [tw93/Pake](https://github.com/tw93/Pake) — NEW ⭐ 53,408（+2,398）| Rust | 信源：中

**它是什么**: 一个命令行工具，一行命令就能把任意网页打包成独立的桌面应用。基于 Tauri 用系统原生 WebView（Windows 的 WebView2 / macOS 的 WKWebView / Linux 的 WebKitGTK），产物只有约 5MB，比 Electron 小近 20 倍。

**解决什么问题**: 很多人把 ChatGPT、微信网页版、Twitter 等高频网页常驻在浏览器标签页里，标签页越开越多还吃内存。Pake 把这些网页剥离成带独立窗口、图标、启动项的桌面 app，启动快、内存省，还能加快捷键、沉浸窗口、去广告。相比 Electron 方案省下大量资源。

**谁在用、怎么用**:
- 场景 1（个人效率）: 把高频 SaaS（ChatGPT/微信/Twitter）变成独立 app，解决标签页溢出。
- 场景 2（二次打包）: 10.7K forks（fork:star 高达 1:5），大量用户 clone 来自定义打包自己的 app，是真实使用的强信号。

**社区怎么评价**:
- 正面: 核心卖点高度一致——原生 WebView、5MB 产物、一行命令、支持快捷键/沉浸窗口/拖拽/样式定制。
- 负面: 首次打包需配置环境且较慢（后续构建快）；Windows 路径过长构建失败（#1230 LNK1104）；CLI 选项文档不全（#1228）。
- 健康度: stars/subscribers ≈ 205:1（成熟工具结构性高比）；open issues 仅 5，仓龄约 3.7 年，最近 push 2026-06-18，维护活跃。

**风险与争议**:
- **dependency**: 依赖各平台系统 WebView，行为有差异（Windows 构建坑较多）。
- **vanity**: HN/Reddit 几乎无一手讨论（HN 多次提交均 2 分/0 评论），外部口碑主要来自博客/媒体转载——传播热度 > 独立社区 grounding。

**技术备注**: Rust + Tauri，系统原生 WebView，MIT。知名老项目首入 tracker。

### [pppscn/SmsForwarder](https://github.com/pppscn/SmsForwarder) — NEW ⭐ 26,416（+130）| Rust | 信源：强

**它是什么**: 一个 Android app，监控手机的短信、来电、APP 通知，按你设定的规则转发到钉钉/企业微信/飞书机器人、邮箱、Bark、Telegram、Server酱、webhook 等渠道；还能远程控制手机发短信、查通话/话簿/电量。

**解决什么问题**: 很多人有一台闲置旧手机或副卡，但收验证码、运营商通知还得随身带。SmsForwarder 把旧机插卡当「短信网关」，验证码和通知自动转发到主力设备或群里；iPhone 用户也能用 Bark 接收安卓卡的短信。

**谁在用、怎么用**:
- 场景 1（短信网关）: 旧机插卡放家里，远程收验证码/运营商通知。
- 场景 2（RPA 自动化）: 影刀 RPA 社区有专文教用它转发验证码做自动化授权。
- 场景 3（保活痛点）: Issues 集中在一加/中兴/MIUI 杀后台导致掉线、厂商 ROM 的「验证类短信保护」导致验证码收不到，需手动关保护并加白名单。

**社区怎么评价**:
- 正面: 渠道支持极全，中文社区（异次元、影刀 RPA）多篇实测教程，真实使用群体大。
- 负面: 厂商 ROM 保活/验证码保护带来的工程困扰是高频 issue。
- 健康度: stars/subscribers ≈ 211:1（成熟工具结构性高比）；forks 3,270，open issues 仅 20 且活跃，仓龄 5+ 年，非 archived。

**风险与争议**:
- **privacy**: 核心功能即转发含验证码的短信，官方 wiki 自带风险提示「转发验证码可能损害账户安全」；用户需关闭系统验证码保护才能用，等于主动削弱设备安全。
- **compliance**: 远程读短信/通话/话簿、控制发短信，自用合规，但代收他人短信涉法律灰区，存在被灰产滥用面。
- **dependency**: 转发链路依赖第三方机器人/推送服务。

**技术备注**: Android（Kotlin 为主，trending 语言标记为 Rust 实为统计误差），V3.0 新增主动控制服务端/客户端。知名工具首入 tracker。

## 如何理解今天的热度信号

今天榜单的可信度分层比较清晰，可按三档理解：

- **真信号（机构/真实使用足迹）**: turso（HN 多帖 + 100+ 贡献者）、timesfm（Google 背书）、Pake/SmsForwarder（高 fork + 中文社区实测）、penpot/insomnia/twenty（成熟开源工具回流）——这些项目的关注度有扎实社区足迹支撑。
- **需结合足迹判断**: palmier-pro、OpenMontage、codebase-memory-mcp、headroom——star 增速明显快于可见的独立社区讨论，仓龄短或营销话术浓，建议点开 Issues/HN/Reddit 再判断价值。
- **收藏型热度**: awesome-artificial-intelligence——awesome-list 的 star 反映收藏热度而非工程采用，不宜与软件项目同等解读。

一个通用提醒：成熟工具（Pake/SmsForwarder/turso）的 stars/subscribers 比值天然偏高（200:1 以上），这是结构性现象，不应单凭比值判定 vanity；真正需要警惕的是「短龄 + 高增速 + 弱社区足迹 + 营销话术」的组合。
