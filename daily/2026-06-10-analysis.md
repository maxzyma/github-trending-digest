## 本日主线

**主线一：agent/skill 仍占多数席位，但今天榜单明显"去 agent 化"——CV、边缘感知、医疗 AI 三个非 agent 方向同日浮现。** 过去两周榜单几乎被 agent/skill 一条赛道吃满，今天结构松动了：计算机视觉出现双仓同框——roboflow/supervision（rank 3，↑9，今日 +733）和 opencv/opencv（rank 4，RE），一个是 CV 工具库、一个是 CV 事实标准底层；再加上边缘感知的 francescopace/espectre（WiFi CSI 隔空人体检测）和医疗 AI 的 maziyarpanahi/openmed（本地医疗 NER 模型集），今天有 4 个项目明确不属于 agent/skill 赛道。更值得注意的是，**3 个真 NEW（system-prompts-and-models-of-ai-tools、openmed、espectre）没有一个是"又一个 agent skill 包"**——上榜新弹药从 agent 工具转向了"agent 的元资料 / 垂直领域 AI / 硬件感知"。读者视角看，这是连续多日 agent 独大后的第一个明显分流日。

**主线二：agent/skill 生态自身进入回流盘整，没有新黑马。** 昨日同框的两大官方仓 google/skills、以及 Agent-Reach / PAI / mempalace / CopilotKit / claude-howto 今天集体掉榜；回补上来的是 addyosmani/agent-skills（RE，最后观测 5/18）和 AiToEarn（RE，6/08 掉榜后回流）。在榜的 pm-skills（↓3）、career-ops（↓5）、openai/plugins（↓4）排名普遍回落，last30days-skill（=，今日仍 +3,191 居增量第一）和 turbovec（=）守住前二。整个 agent 板块今天是"存量项目榜内浮沉"，无跨赛道新冲击——与主线一的非 agent 方向浮现形成对照：榜单的新鲜度今天来自 agent 赛道之外。

**主线三：榜单解读提醒 — "工具仓 / 资料仓"的高 stars/subscribers 比值，要看真实足迹在不在 GitHub 之外。** 今天几个高比值项目恰好提供了"比值高但不等于空心"的清晰样本：openmed 100:1，但真实足迹在 HuggingFace（模型 2,970 万次下载、被 HF《2026 春季开源 AI 报告》列为引用最多组织 #1），GitHub repo 只是入口；system-prompts 84:1，是"资料档案库"形态，84:1 对档案站属正常区间；espectre 107:1，却有 HN Show HN 215 分 / 50 评 + Home Assistant 官方论坛热议 + Espressif 官方背书。判断这类项目不能只盯 GitHub 的 stars/subs，要追问"它的真实使用足迹是不是落在别的平台"。真正需要谨慎解读的仍是另一类——career-ops 今日 +1,110（增量第二）、比值约 250:1，HN/Reddit/X 三处仍零有机讨论，外部声量集中在作者可控渠道，这类"高比值 + 短龄 + 零社区足迹 + 营销集中"的组合才是 vanity 判据。

### [x1xhlol/system-prompts-and-models-of-ai-tools](https://github.com/x1xhlol/system-prompts-and-models-of-ai-tools) — NEW ⭐ 139,188（+79）| TypeScript | 信源：中

**它是什么**: 一个收集主流 AI 编码工具"完整系统提示词 + 内部工具定义"的档案仓——Cursor、Claude Code、Devin、Windsurf、v0、Manus、Augment、Replit 等 28+ 个工具，把它们提取/泄露出来的内部 prompt 公开归档。开发者把它当成"比官网更真实的产品行为文档"来读，用来逆向这些 agent 到底是怎么工作的。维护者是 16 岁西班牙开发者 Lucas Valbuena（@NotLucknite）。

**解决什么问题**: 之前想知道某个 AI 编码工具内部是怎么组织 prompt、怎么定义工具调用的，只能靠猜或零散的截图。这个仓把各家的系统提示词集中归档，prompt engineering 研究者可以直接对比——比如 Devin 的 "code archaeology" 逐行证据策略、Windsurf 把工具 API 以 TypeScript 类型暴露、Cursor/v0 共用的 `// ... existing code ...` lazy-edit 策略，都能横向看到差异。

**谁在用、怎么用**（来自 Issues/外部分析）:
- 场景 1: prompt 研究者把它当真实文档读，逆向各 agent 的行为模式与工具定义设计。
- 场景 2: 社区持续追新——issue #393/#453/#454 在补 Google Antigravity IDE 提示词提取（约 1 万 token）、Gemini 3.5、Perplexity 2026 以及中文 prompt + 转 Markdown 格式。每有新 AI 工具发布就追一批，制造反复曝光（这也是它今日 13.9 万 stars 基数下仍能回到榜单的原因，今日增量本身只有 +79）。

**社区怎么评价**:
- 正面: 被当作真实文档广泛引用，揭示各工具的设计差异；有第三方深度分析（Augment Code 等）。
- 负面: authenticity 争议——内容是"泄露 + 逆向"混合，有 Medium 分析（《Fake it till you prompt it》）指出提示词内部存在不一致，怀疑部分内容真伪或被模型幻觉污染；维护者借此引流自家 prompt 提取审计服务 ZeroLeaks（README 挂链），有商业化色彩。
- 健康度: 139,188 stars / 1,653 subscribers（约 84:1，对"资料档案库"形态属正常区间，非工具型 vanity）；28 contributors、约 489 commits，最近提交 2026-06-09，PR 持续合入，维护活跃。

**风险与争议**:
- **compliance**: 发布提取的专有系统提示词大概率违反各工具 ToS（提取方式常为 prompt injection）；提示词的版权归属属法律灰区，无判例。
- **authenticity**: 逆向内容真伪存疑，"当文档读"存在被误导风险。
- **dependency**: 内容时效靠不断追新维持，旧版提示词会过时。

**技术备注**: TypeScript；GPL-3.0；覆盖 28+ AI 编码工具的系统提示词与内部工具定义。

### [maziyarpanahi/openmed](https://github.com/maziyarpanahi/openmed) — NEW ⭐ 1,890（+191）| Shell | 信源：中

**它是什么**: 开源的本地化医疗 AI 工具集，作者 Maziyar Panahi 是 Spark NLP 主创、CNRS 资深 AI 工程师（HuggingFace 知名人物，非无名作者）。核心不是单一大模型，而是 1,000+ 个专用医疗 NER 小模型 + 一个 Python 库（`pip install openmed`）+ Swift/MLX 端侧套件——把临床文本一行代码转成结构化实体（疾病 / 药物 / 肿瘤 / 基因等），并做 HIPAA Safe Harbor 18 项 PII 脱敏。

**解决什么问题**: 医院和生命科学机构处理临床文本时，既要抽取结构化信息，又不能把病人数据传给外部 API。openmed 主打"全程本地、不出设备"——可跑 CPU / CUDA / Apple Silicon 甚至 iPhone，Apache-2.0，让机构在自己的 VPC 或端侧完成实体抽取与去标识。有 arXiv 论文（2508.01630），在 12 个公开数据集中 10 个刷到 SOTA micro-F1。

**谁在用、怎么用**（来自 Issues/HuggingFace）:
- 场景 1: 医院 / 生命科学机构在院内或 VPC 内做临床文本实体抽取与去标识，端侧 iPhone 跑 NER。
- 场景 2: issue 量少但真实——#27 PII 讨论（10 评论，外部用户参与）、#22 请求 REST API、#20 问是否支持 CoT；HuggingFace 评论区有护理行业用户称对提升数据管理 / 隐私标准有帮助。

**社区怎么评价**:
- 正面: 真实足迹在 HuggingFace 而非 GitHub——模型累计 2,970 万次下载、PyPI 55 万次、被 HF《2026 春季开源 AI 报告》列为被引用最多的组织 #1。
- 负面: GitHub 社区互动确实薄（issues / HN / Reddit 均弱），外部声量主要靠作者 X 自传播。
- 健康度: 1,890 stars / 19 subscribers（约 100:1，比值偏高，但属"工具仓"形态——真实足迹在 HF 不在 repo）；open issues 仅 9；贡献者仅 2 人（典型单人主导），仓库昨日刚 push，维护勤。

**风险与争议**:
- **compliance**: "HIPAA-aware 脱敏"是自我宣称，实际合规需机构自验，不能仅凭工具背书。
- **accuracy**: 医疗 NER 准确度直接关乎临床安全，README 宣称 SOTA 但社区独立第三方验证薄弱。
- **bus_factor**: 仅 2 贡献者，强单人依赖。

**技术备注**: Python 库 + Shell 脚手架；Apache-2.0；1,000+ 医疗 NER 模型；支持 CPU/CUDA/Apple Silicon/iPhone（MLX）。

### [francescopace/espectre](https://github.com/francescopace/espectre) — NEW ⭐ 8,226（+134）| —— | 信源：强

**它是什么**: 用一块 ESP32 监听 WiFi 信道状态信息（CSI）做"隔空人体感知"——人在房间走动会扰动路由器与 ESP32 之间的 WiFi 信号，系统据此判断有没有运动 / 有没有人。无摄像头、无麦克风，通过 ESPHome 原生接入 Home Assistant，在面板里呈现为一个 binary motion sensor。

**解决什么问题**: 传统存在检测要么用 PIR（需视线、对静止的人无感），要么用 mmWave 雷达（更贵）。espectre 只要一块约 €10 的 ESP32、不用改路由器、能穿干墙（2.4GHz），还对静止的人也能感知，主打"便宜 + 隐私友好 + 无死角"。算法走统计 / 信号处理路线（NBVI 自动选子载波，F1>96%），v2.5 起加了实验性的端侧神经网络检测器。

**谁在用、怎么用**（来自 Issues/HN/HA 论坛）:
- 场景 1: 智能家居玩家用它替代 PIR 做无死角 / 静止人存在检测，驱动灯光自动化和安防场景。
- 场景 2: 社区有活跃 PR 共建——#112 加呼吸带通滤波做静止存在检测、#111 时序平滑投票、#129 支持 ESP32-C5/C6 UDP 满吞吐；HA 论坛有"ESP32-S3 上测试，惊呆了"的实测反馈。

**社区怎么评价**:
- 正面: HN 口碑强——Show HN（2025-11-17）215 分 / 50 评；Home Assistant 官方论坛热议；Espressif 官方 esp-csi 仓把它列为 community project（背书）。
- 负面: ESP32-C3 上报 `ENOMEM` / Send error（作者称不影响检测）；阈值 / 自动校准一致性问题（断电后 threshold 显示 unknown）；ESP8266 不支持；准确率依赖 3-8m 摆位与环境，混凝土墙会衰减。
- 健康度: 8,226 stars / 77 subscribers（约 107:1，比值偏高但符合 Show HN 爆款形态）；open issues 仅 9，pushed 2026-06-09 活跃；作者 226 次提交、其余多为 1-2 次，单核心维护者。

**风险与争议**:
- **privacy**: WiFi 隔空感知人员活动，虽无影像，但被动探测室内人体移动本身存在感知监控争议。
- **bus_factor**: 单一核心维护者。
- **stability**: 校准 / 内存告警，端侧 ML 检测器仍是实验性，准确率受摆位与墙体材质影响。

**技术备注**: ESP32 固件 + ESPHome 集成；GPL-3.0；WiFi CSI + 统计算法（v2.5 起含实验性端侧 NN）。
