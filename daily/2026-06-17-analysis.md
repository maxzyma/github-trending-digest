## 本日 NEW 项目点评

### [n0-computer/iroh](https://github.com/n0-computer/iroh) — NEW ⭐ 9,310（+334）| Rust | 信源：强

**它是什么**: 一个用 Rust 写的 P2P 网络栈。传统网络要靠 IP 地址连接，但 IP 会变、会被 NAT 挡住；iroh 让你用一个公钥（NodeId）直接"拨号"对方设备，自动完成 NAT 打洞，连不通时回退到中继服务器。可以理解为"给应用嵌入的、按身份而非地址连接的底层网络管道"。

**解决什么问题**: 没有它之前，做 P2P/跨设备同步的开发者要么自己折腾 NAT 穿透、STUN/TURN，要么依赖庞大的 libp2p。iroh 把"两个设备直连"这件事收敛成一个库：给对方公钥即可连接，覆盖从 ESP32 嵌入式到服务器的全场景，附带文件传输（blobs/BLAKE3）、gossip pub-sub 等上层能力。

**谁在用、怎么用**（来自 Issues/HN/X）:
- WebTransport（#3750）与 ESP32 嵌入式（#3990）支持需求活跃，说明被浏览器与嵌入式场景实际采用。
- Holochain 0.7 集成 iroh 做打洞、Fedimint/Start9 用它去除静态 IP 依赖——多个独立项目真采用，非作者自吹。

**社区怎么评价**:
- 正面: HN Iroh 1.0（6/15）**1,351 分 / 423 评论**登顶，多年多次进前列；X 官方 @iroh_n0 "Dial Keys, not IPs" + 第三方项目背书。
- 负面: #3565 疑似内存泄漏（11 评论）、#4201 relay 连接建立时间被误计入 RTT（14 评论，测量准确性）。
- 健康度: 9,310 stars / 65 subscribers ≈ 143:1（早期基础设施库常见结构）；forks 437、56 贡献者，2022 建仓 4 年迭代 65+ 版本，多人协作非单人。

**风险与争议**:
- **stability**: 1.0 刚发布，relay 路径存在内存泄漏（#3565）与 RTT 测量 bug（#4201），需观察。
- **dependency**: 连不通时回退依赖自建的公共 relay 生态。

**技术备注**: Rust，QUIC 直连，ed25519 公钥寻址，自动 NAT 打洞 + relay 回退。

### [alibaba/zvec](https://github.com/alibaba/zvec) — NEW ⭐ 10,492（+156）| C++ | 信源：中

**它是什么**: 一个轻量、嵌入进程内的向量数据库——被官方类比为"向量数据库界的 SQLite"。不需要单独部署服务端，直接以库的形式嵌入应用，在本地/边缘设备上做向量检索。

**解决什么问题**: 没有它之前，做 on-device RAG 或边缘 AI 检索要么背上 Milvus/Qdrant 这类需独立部署的服务，要么自己拼 faiss。zvec 把向量检索做成进程内嵌入式组件，支持 hybrid search、WAL 持久化、多进程读单进程写，适合 notebook/CLI/边缘设备等无服务器场景，并提供 Python/Go/Rust/Flutter 多语言 SDK。

**谁在用、怎么用**（来自 Issues/HN）:
- #380「Export DB」(6 评论)、#61「Go SDK / Go bindings」(4 评论) 等功能诉求型 issue 活跃，是真实使用意向信号。
- 底层是阿里 Proxima 引擎（已在淘宝搜索/支付宝刷脸/优酷生产多年），zvec 是其嵌入式重封装——非空壳。

**社区怎么评价**:
- 正面: HN 主帖「Zvec: A lightweight, fast, in-process vector database」**226 分 / 45 评论**，社区关注扎实。
- 负面: issue 量级偏薄（58 open），未见高热崩溃/数据丢失类抱怨，但也说明用户基数尚未转化为提问。
- 健康度: 10,492 stars / 56 subscribers ≈ 187:1（成熟阿里系结构性偏高），forks 607、23 贡献者，2025-12 半年新仓。

**风险与争议**:
- **vanity**: stars/subscribers 比偏高 + 二手测评博客（"SQLite of Vector DB" 话术复用）密集 + X/Reddit 零有机讨论，传播热度强于独立社区验证。
- **stability**: 半年新仓，性能宣称（>8000 QPS、2× ZillizCloud）为官方自报 benchmark，缺第三方复现。

**技术备注**: C++ 实现，底层 Proxima 引擎，支持 hybrid search / WAL，多语言 SDK。

### [Universal-Debloater-Alliance/universal-android-debloater-next-generation](https://github.com/Universal-Debloater-Alliance/universal-android-debloater-next-generation) — NEW ⭐ 7,334（+146）| Rust | 信源：中

**它是什么**: 一个跨平台 GUI 工具（Rust 写），通过 ADB 给未 root 的 Android 手机"瘦身"——禁用/卸载三星、小米等 OEM 预装应用和广告软件，提升隐私、安全与续航。是已停更的原版 UAD（0x192/UAD）的联盟接棒分支。

**解决什么问题**: 没有它之前，普通用户想去掉手机厂商预装 bloatware 要么 root（有风险），要么手敲 ADB 命令逐个禁用。UAD-NG 把这套操作做成图形界面 + 社区维护的可删包列表，接上电脑即可批量免 root 去预装。

**谁在用、怎么用**（来自 Issues/HN）:
- 社区持续补包列表，#583「Safe removal list」(20 评论) 是最热 issue。
- 真实场景：连电脑跑 ADB，禁用/卸载 OEM 预装 + Facebook/Twitter 等，提升隐私续航。

**社区怎么评价**:
- 正面: 长期维护、跨平台、100+ 贡献者联盟治理；HN 同主题「Debloat non-rooted Android devices」(2024) 303 分 222 评论高热背书；Chris Titus Tech 等创作者正面背书。
- 负面: 核心痛点是误删变砖——#1400 小米国行 ROM、#1311 三星 recommended 列表均导致 bootloop，即包分类对国行/三星 ROM 不够安全。
- 健康度: 7,334 stars / 38 subscribers ≈ 193:1（工具类正常），315 forks、246 open issues，2023-10 建仓持续活跃。

**风险与争议**:
- **stability**: 多起 bootloop issue 说明"recommended"分类对国行/三星 ROM 不安全，需谨慎按推荐删除。

**技术备注**: Rust + iced GUI，依赖 ADB，无法移除固件级 vendor 服务（README 已声明）。
