### [NVIDIA/SkillSpector](https://github.com/NVIDIA/SkillSpector) — NEW ⭐ 2,711（+319）| Python | 信源：中

**它是什么**: NVIDIA 开源的 AI agent skill 安全扫描器。你从网上装一个 Claude Code / Codex / Gemini CLI 的 skill 之前，它帮你回答"这个 skill 安全吗"——静态分析 + 可选 LLM 语义评估，覆盖 64 种漏洞模式（prompt 注入、凭据窃取、供应链投毒等 16 类），输出 0-100 风险评分和 SARIF 报告。

**解决什么问题**: skill 本质是"别人写的指令 + 脚本，直接喂进你的 agent"，安装即信任。Snyk 的 ToxicSkills 研究发现 ClawHub 市场上 13.4% 的 skill 存在严重安全问题，但此前用户除了肉眼读 SKILL.md 没有任何检查手段。SkillSpector 把"装前扫一遍"做成了标准动作。

**谁在用、怎么用**（来自 Issues/合作方）:
- 场景 1: ClawHub 官方合作——每个上架 skill 都过 SkillSpector 扫描；NVIDIA 自家 Verified Agent Skills 目录的发布验证流水线在用。
- 场景 2: 外部开发者提交实质性安全加固 PR——#19 限制文件读取防 DoS、#3 修 Anthropic provider 兼容、#27/#28 修 MCP/供应链误报，工程参与真实。

**社区怎么评价**:
- 正面: 方向公认刚需，有 OpenReview 配套论文（pre-publication security control），厂商背书 + 市场采用双实证。
- 负面: #14 "False Report" 投诉对 clawhub gitlab-agent skill 的误报；第三方交叉研究显示 VirusTotal/静态分析/SkillSpector 三者检出重叠率 <10.4%，误报/漏报校准是核心争议。
- 健康度: 2,760 stars / 15 subscribers（约 184:1）；216 forks；2026-03 创建，push 活跃；贡献者 API 仅显示 4 人（NVIDIA 内部 squash 流程）。HN/Reddit 零讨论，以 NVIDIA 的名头看略反常；X 传播偏 KOL 转述。

**风险与争议**:
- **误报校准**: 多扫描器交叉重叠率 <10.4%，#14 误报争议未闭环——扫描结论当参考，不当裁决。
- **stability**: 3 个月项目，检测规则与 LLM 评估的校准仍在早期。

**技术备注**: Python；静态分析 + LLM 语义评估双引擎；支持 Claude Code / Codex CLI / Gemini CLI skill 格式；输出 SARIF；与 ClawHub 上架流水线集成。

### [kenn-io/agentsview](https://github.com/kenn-io/agentsview) — NEW ⭐ 1,676（+114）| Go | 信源：强

**它是什么**: Pandas 作者 Wes McKinney 做的"AI 编程 agent 会话分析台"。Claude Code、Codex、Cursor 等 27 种 agent 都会在你本机留下会话文件，agentsview 把它们索引进 SQLite，让你浏览/全文搜索历史会话、看 token 成本看板、生成 AI 编码活动日报——全部本地优先，自称比 ccusage 快 80-220 倍。

**解决什么问题**: 重度 agent 用户的两个日常问题此前没有顺手答案："我上周让 agent 干了什么、那段对话在哪"（会话散落在各工具私有目录里无法检索）和"我今天花了多少 token 钱"（ccusage 只覆盖 Claude Code 且慢）。agentsview 把跨工具的会话资产变成可查询的本地数据库。

**谁在用、怎么用**（来自 Issues/博客）:
- 场景 1: 跨 agent 的 token 花费日报与会话全文检索复盘；Simon Willison 6/09 博文公开推荐（"been really enjoying"），已有衍生项目 Agentlore 基于它构建团队日志。
- 场景 2: 需求流活跃——#413（wesm 本人，Cursor state.vscdb 导入）、#529（antigravity-cli 支持）；issue 总编号 4 个月内已到 647。

**社区怎么评价**:
- 正面: 作者名人效应 + Simon Willison 背书；wesm 主导（337 commits）但有 10+ 真实外部贡献者（mariusvniekerk 38 commits 等）。
- 负面: #562 非预期弹出 "Authentication Required"（8 评论）；#224 Gemini insights 生成失败；HN 多次提交全部 1-4 分零评论，Reddit 无讨论——传播集中在 KOL 渠道。
- 健康度: 1,702 stars / 4 subscribers（约 425:1，全榜最高，但疑似 wesm 个人仓迁入 kenn-io org 时 subscribers 重置，不宜直接判 vanity）；89 open issues；2026-02-19 创建，今日仍有 push。

**风险与争议**:
- **数据失真**: subscribers=4 疑为仓库迁移重置产物，425:1 比值不构成 vanity 证据，但需标注。
- **stability**: 认证弹窗（#562）、Gemini insights 失败（#224）等早期 bug 仍在。
- **bus_factor**: wesm 单人主导提交量，项目节奏强依赖作者。

**技术备注**: 本地优先；SQLite 索引；覆盖 27 种 agent 会话格式；`curl | bash` 安装（介意者可从源码装）；ccusage 替代定位。

### [hexo-ai/sia](https://github.com/hexo-ai/sia) — NEW ⭐ 1,326（+199）| Python | 信源：中

**它是什么**: Hexo Labs 开源的"自我改进 AI"框架——让 AI agent 在 benchmark 任务上自动迭代改进自己：既改 agent 的脚手架（prompt、工具逻辑），也用 LoRA/PPO 直接更新模型权重。配套论文 arXiv:2605.27276，宣称在 LawBench 上 70.1% 超 SOTA。

**解决什么问题**: 让 agent 变强目前靠人工调 prompt、调工具链、攒数据微调。sia 想把这个循环自动化：agent 跑任务 → 评估器打分 → agent 自己改自己 → 再跑。如果成立，它是"用计算换工程师调优时间"的路径。

**谁在用、怎么用**（来自 Issues）:
- 场景 1: 研究者跑 4 个内置 benchmark（gpqa/lawbench/chess/titanic）并修 evaluator——#31/#34 修 evaluator 输出 bug，提交者质量高但全部 0 评论。
- 场景 2: HuggingFace 的 NielsRogge 在 #35 请求释出 SIA-W+H 权重——论文模型权重尚未开源，复现受限。

**社区怎么评价**:
- 正面: X 上官方 @hexoai 发布 + KOL 放大（@rohanpaul_ai 等，有"打败 Karpathy 的 autoresearcher"式标题）；issue 提交者多为有名有姓的研究者。
- 负面: 质疑声明确——Goodhart 风险（固定 verifier 被双重优化）、"recursive wishful thinking"、缺安全护栏（eval/版本/回滚）；#36 指出 held-out 参考答案泄漏给 feedback agent，benchmark 完整性存疑且未闭环。
- 健康度: 1,348 stars / 8 subscribers（约 169:1）；162 forks；2026-03-25 创建不到 3 个月，push 活跃；前 4 名贡献者全是 Hexo 员工。HN 仅 3-4 分零评论，Reddit 无讨论。

**风险与争议**:
- **vanity**: 传播靠 BusinessWire 通稿 + 科技媒体转载 + X KOL，HN/Reddit 零有机讨论。
- **marketing**: 通稿宣称"加速超智能 350X"，该数字不在论文中。
- **benchmark 完整性**: #36 答案泄漏问题未关闭，LawBench 70.1% 的可信度待第三方验证。
- **dependency**: SIA-W+H 权重未释出，核心结果外部不可复现。

**技术备注**: Python；LoRA/PPO 权重更新 + 脚手架自改写双通道；内置 gpqa/lawbench/chess/titanic 四个 benchmark；arXiv:2605.27276。

## 如何理解今天的热度信号

今天的对照组是"高比值的四种成因"。SkillSpector（184:1）的比值来自大厂官方仓的结构性常态——subscribers 低但有生产采用和外部 PR 兜底；agentsview（425:1）的比值是**数据工件**——仓库迁移重置了 subscribers，用原始比值判断会得出完全错误的结论；sia（169:1）和 zhangxuefeng-skill（190:1）的比值则真实反映了"传播渠道在作者可控范围内"（通稿+KOL / 粉丝盘+玩梗）。同一个指标，四种成因：结构性、工件性、营销性、圈层性。比值永远只是触发器，要分辨成因必须看三件事——外部贡献者是否真实存在、生产采用是否可验证、传播是否独立于作者渠道。另外提一句 agency-agents：停更两个月净增 7,700 星，"维护状态"与"热度"是两条独立曲线，点开项目前先看 pushed_at 是今天最便宜的避坑动作。
